## 0. 한 줄 요약

RAGBuilder(외부 도구가 파이프라인을 소유해 자체적으로 탐색)를 버리고,
**"진단이 실측한 값 → 파라미터 목표값을 계산"** 하는 방식으로 간다.
계산이 안 되는 파라미터만 소수 후보를 실제 평가(sweep)하고, 탐색은 우리 파이프라인
루프가 주도한다. 이 방식은 **더 정확하고(추측 아님), 더 싸고(후보 최소), 설명
가능(왜 이 값인지 근거 제시)** 하다.

---

## 1. 배경 — 왜 RAGBuilder를 버리는가

원래 계획은 RAGBuilder로 top_k·chunk_size 등을 튜닝하는 것이었으나, 무겁고 우리
프로젝트와 통합이 어려웠다. 근본 원인은 성능이 아니라 **제어권 방향(control
inversion)** 이다.

- **RAGBuilder가 원하는 것**: "너희 파이프라인을 나에게 넘겨라. 내가 내부에서 여러
  config를 돌려보고 best_config를 돌려주겠다." → 자체 surrogate 파이프라인을 만들어야
  함 = 무겁고 통합 난이도 높음.
- **Agent Doctor가 원하는 것**: 우리는 이미 평가 루프
  (`Optimize → Index → Eval → Optimize`)를 갖고 있다. 필요한 건 "다음에 어떤 값 하나를
  시도할지"만 알려주는 가벼운 두뇌다.

> **교훈(다른 도구 선택 시 제1원칙)**: 우리 루프가 **부를 수 있는(드라이브당하는)**
> 도구여야 한다. "파이프라인을 달라"는 도구는 RAGBuilder처럼 또 실패한다.
> (참고: Optuna 등은 ask-and-tell 모드로 이 조건을 만족할 수 있으나, 아래 3~4절의
> 이유로 현 단계에선 직접 구현이 더 낫다.)

---

## 2. 핵심 아이디어 — 측정이 탐색을 대체한다

일반적인 하이퍼파라미터 튜닝은 "어떤 knob이 문제인지 모르니 다 돌려보자"는 **무지성
탐색**이다. 그러나 Agent Doctor는 그 앞에 **진단(Eval → Finding)** 이 있어서, "이 문제는
top_k 때문"이라고 knob을 이미 찍어준다.

즉 우리가 풀 문제는 "여러 knob 중 뭘 돌릴까?"가 아니라 **"진단이 찍어준 이 knob의 값을
뭘로 할까?"** 이다. 그리고 이 값은 대부분 **진단이 이미 잰 숫자에서 계산**할 수 있다.

**예시 (top_k):**
- 나쁜 방식: top_k=5인데 못 찾았으니 ×2=10으로 늘려보자 (추측)
- 근거 방식: Eval이 "정답 청크가 **11위**에 있었다"를 실측했다면 → `top_k = 11`.
  ×2(=10)는 여전히 못 잡는다. 11이라는 값은 대중값도 추측도 아니라 **"이 파이프라인이
  이 질문에서 실제로 정답을 11위에 뒀다"는 측정**에서 나온다.

이것이 Agent Doctor의 차별점("왜 실패했는지 설명")을 **값 선택 단계까지** 밀어붙인
것이다.

---

## 3. 통합 모델 — 측정은 후보를 좁히고, 실측이 결정한다

> ⚠️ **정정**: 초안에는 라벨을 "계산 / sweep / 토글" **3층으로 분류**하는 표가 있었으나
> **폐기했다.** 그 분류는 불필요했고 두 가지를 뒤섞고 있었다.
>
> | 질문 | top_k | chunk |
> |------|-------|-------|
> | 값이 측정으로 **확정**되나? (인식론) | ❌ 아니오 | ❌ 아니오 |
> | 여러 값 시도가 **비싼가**? (경제학) | 비쌈 | 비쌈 |
>
> 초안은 top_k를 "①이 된다"고 보고 계산 층에 넣었지만, **①은 어느 축도 안 된다.**
> probe 1개면 "gold가 11위 → top_k=11"이 정확하지만, probe 10개가 서로 다른 값을
> 요구하면 정답이 하나로 안 나온다 — 노이즈와 커버리지의 **교환비**를 알아야 하는데
> 그건 계산이 아니라 측정의 영역이다.

### 실제 모델

```
측정  → 원칙 있는 후보 1~3개로 범위를 좁힌다   (최종 답을 내는 게 아님)
sweep → 그중 승자를 실제 평가로 결정한다       (심판은 항상 overall_score)
```

**Planner는 파라미터 종류와 무관하게 `{canonical_경로: [후보값...]}` 하나의 형식으로만
전달한다.** 그 후보를 어떻게 실행할지(실제 sweep / 싼 사전검증 / 1회 적용)는 **Optimizer가
경로별 정책으로 결정**한다. Planner가 분류를 알 필요가 없다.

| canonical 경로 | Optimizer의 실행 정책 | 비싼 Index/Eval 횟수 |
|---|---|---|
| `retriever.top_k` | 후보별 실제 Eval 후 best 선택 | 후보 수만큼 |
| `chunker.chunk_size` / `chunk_overlap` | **싼 기하 사전검증**으로 예상 best 하나만 선택 | 1회 |
| 단일 토글 | 안전검사 후 적용 | 1회 |

후보가 1개면 sweep 할 게 없어 `rules` 로 1회 검증하고, 2개 이상이면 `internal` backend가
방문에 걸쳐 sweep 한다. 자세한 정책은 `OPTIMIZER_IMPLEMENTATION_PLAN.md` §1 참고.

### 머신러닝(Bayesian/Optuna)을 왜 안 쓰나

ML 최적화는 "많은 값을 **싸게** 평가"할 수 있을 때 빛난다. 우리 평가는 **비싸고(파이프라인
전체 재평가) 횟수가 적어** ML이 surrogate를 학습할 표본이 부족하다 — 이점이 거의 안 난다.
또 ML은 블랙박스라 "왜 이 값인지" 설명을 못 해 제품 철학과 어긋난다. 측정으로 후보를
좁히고 실측으로 고르는 쪽이 더 정확하고 싸고 설명 가능하다.

## 4. 파라미터별 근거 메커니즘

### 4-1. top_k — 한계비용 무릎 분석 (구현: `planner._knee`)

- **근거 데이터**: 필요한 gold **개수**(`len(finding.affected_chunks)` — 오늘 측정됨),
  놓친 gold의 **순위**(Eval 계약 대기 — §5·§8).
- **여러 probe 집계**: **평균·최댓값을 쓰지 않는다.** 평균은 이상치에 끌려가고(아래
  예시에서 16.4), 최댓값은 노이즈 폭발이다. 대신 **커버리지 곡선의 한계비용**을 본다.

필요값 `[3,4,4,5,6,7,8,12,15,100]`이면:

| top_k | 커버 | probe 1개당 든 비용 |
|---|---|---|
| 8 | 7/10 | **1.0** ← 여기까지 효율적 |
| 12 | 8/10 | **4.0** ⚠️ 급등 |
| 15 | 9/10 | 3.0 |
| 100 | 10/10 | **85.0** 💀 |

무릎 = 비용이 급등하기 직전(=8). **이상치 100이 배제되는 이유가 "통계적으로 이상해서"가
아니라 "하나 더 커버하려고 85를 더 쓰는 게 밑지는 장사라서"** 라는 점이 중요하다.
백분위(P80 등)보다 나은 이유: "왜 하필 80%인가"엔 답이 없지만, "질문 1개당 청크 몇 개까지
낼 것인가"는 실제 비용·노이즈로 **보정 가능한** 질문이다. (probe 10개로는 P80과 P85가
통계적으로 구별되지도 않는다 — 허위 정밀도다.)

**무릎은 최종 답이 아니라 sweep 구간의 시작점이다.** 임계값(`_MAX_STEP_PER_PROBE`)은
교환비에 대한 추측이므로, 맞았는지는 실측이 확인한다:

```
무릎 아래(6,7): 후보 제외 — 무릎이 지배(커버리지 높고 노이즈 차이 작음)
무릎 위(12,15): sweep 대상 — '밑진다'는 게 추측이라 실측으로 확인
→ 후보 [8, 12, 15] → optimizer="internal" → overall_score 가 승자 결정
```

무릎이 전부 커버하면 후보가 1개 → `rules` 로 1회만 검증(비싼 평가 절약).

- **이상치가 안전범위를 넘을 때**: gold가 200위라는 건 "top_k가 작아서"가 아니라 "리트리버가
  그 청크를 아예 못 알아본 것" = **다른 처방(임베딩/reranker/hybrid) 문제**다. top_k를
  200으로 올리면 노이즈·비용 폭발. 안전범위 `top_k ≤ 20`(optimizer의 `DEFAULT_CONSTRAINTS`)이
  이런 근거값을 자동으로 걸러내므로, 이상치는 "처리할 버그"가 아니라 **"top_k가 답이
  아님을 알리는 신호"** 로 취급하고 자연히 다른 처방으로 넘긴다.

### 4-2. chunk_size

- **원리**: 한 청크가 답 하나를 온전히 담되 잡음은 최소로. 답 span이 250자인데
  chunk_size가 1024토큰이면 잡음 과다(overchunking/too_long), 답 span이 800자인데
  chunk_size가 256토큰이면 답이 경계에서 잘림(underchunking).
- **계산(이상적)**: 답 span 길이 분포를 측정 → `chunk_size ≈ 85백분위 span 길이 + 마진`.
  방향은 라벨이, 크기는 span 통계가 준다.
- **싼 사전검증 가능성**: 답의 원문 좌표(`gold_char_span`)는 재청킹돼도 안 깨지므로,
  "chunk_size가 X면 이 답이 잘릴까?"를 재청킹·재임베딩·LLM 없이 **기하학적으로 근사**해
  후보를 미리 순위 매길 수 있다(→ 비싼 평가 전에 후보 축소). 단 실제 청커는 문장 경계를
  존중하므로 기하 시뮬은 **근사**이지 정확한 대체는 아니다.

### 4-3. chunk_overlap

- **원리**: 청크 경계가 답을 두 동강 내지 않게. 근거 = **경계 절단 분석**(gold_span이
  청크 경계를 가로지르는지).
- **계산(이상적)**: `overlap ≥ 잘리는 답의 길이`. 상한은 코드가 이미 강제
  (`overlap ≤ chunk_size × 0.4`).

## 5. ⚠️ 지금 Eval이 실제로 재는 것 (데이터 준비 현실)

**중요**: 위 4절의 메커니즘은 설계상 타당하지만, **실제로 데이터가 흐르는 정도는
파라미터마다 다르다.** 코드를 확인한 결과:

| 파라미터 | 근거 데이터 상태 | 필요한 선행 작업 |
|----------|------------------|------------------|
| **top_k (gold 개수)** | ✅ **구현 완료**. `incomplete_enumeration` 이 `len(affected_chunks)` 로 무릎 분석 → sweep 후보 | 없음 |
| **top_k (순위)** | `low_rank`/`missing_gold` 는 "코퍼스에 있나"(불리언)만 재고 **순위는 계산 과정에 있으나 버려짐** | Eval 계약 1개(gold_rank 노출) — **작음** |
| **chunk_size** | 답 텍스트 길이(`ground_truth`)만 있는 **약한 근거**. 정확한 span 위치(`gold_spans`)는 스키마·재계산 기계는 있으나 **RAGAS probe 생성기가 아직 안 채움**(no-op) | `gold_spans` 채우기 — **큼** |
| **chunk_overlap** | **없음**(경계 위치 필요) | `gold_spans` 채우기 필수 — **큼** |

추가로, `chunking_*` 라벨(underchunking/overchunking)은 **Eval에 진단 함수가 아직 없다**
(rules.py에 draft 처방으로만 존재). 즉 Eval이 "청크가 너무 작다/크다"를 아직 진단하지
않는다. chunk 인접 신호는 `too_long_context`의 거친 ablation("절반 줄이면 도움됨 yes/no")
하나뿐이며 이는 방향만 주고 크기 근거는 아니다.

### 해소된 블로커 (2026-07-16)

top_k 처방은 **capability 검사에서 전부 버려지고 있었다**(`skipped`/`unsupported_capability`).
비허용의 근거였던 "파이프라인이 top_k를 읽지 않는다"가 이미 해소된 상태였다 —
`agents/eval/agent.py`가 `index_config["top_k"]`를 읽어 실제 검색에 쓰고 Index도 청크
metadata에 기록한다. 다음을 반영해 잠금을 풀었다.

- `optimizer.DEFAULT_CAPABILITIES["retriever.top_k"] = True`
- `core/state.py`의 `index_config` 기본값에 `top_k: 5` 명시(Eval/Index 폴백값과 동일)
- `rules.py`의 `retrieval_incomplete_enumeration` 을 `ready` 로 승격

> **결론**: **top_k(개수 근거)는 완료**, **top_k(순위 근거)는 Eval 계약 1개**,
> **chunk 근거화는 phase 2**(Eval의 `gold_spans` 채우기 + chunking 신호 추가 선행).

## 6. 비용 통제

파라미터 후보가 많으면 매번 전체 파이프라인을 돌려 비용이 발생한다. 두 겹으로 막는다.

1. **근거가 후보를 줄인다**: 계산되는 라벨(✅)은 목표값 1개만 검증하면 된다(5개 sweep →
   1개). 측정이 탐색을 대체하므로 원리적으로 옳으면서 동시에 싸다.
2. **`confirmed` 게이팅**: Finding에는 `confirmed=True`(확정)와 `False`(예비)가 있다. 예비는
   Eval이 자원(진단 모드/tier) 부족으로 확신하지 못한 진단이다. **확정된 finding에만
   비싼 trial을 쓰고 예비엔 쓰지 않으면** 헛돈을 막는다. (planner가 이 플래그를 반영해야
   한다.)
3. (chunk 한정) **기하 사전검증**으로 비싼 평가 전에 후보를 축소(4-2 참고).

## 7. 안전성 — 무한 진동을 어떻게 막나

우려: top_k↑로 recall은 좋아졌는데 context가 길어져 `too_long_context` 발생 → top_k↓ →
recall 다시 나빠짐 → 무한 반복. 이 greedy(좌표하강)의 진동을 **이미 있는 4중 장치**가
막는다.

1. **단일 점수 keep-gate(가장 중요)**: `history.judge`는 **overall_score가 올라야만
   유지**한다. "A를 고치려다 B를 똑같이 망가뜨리는" 제로섬 교환은 전체 점수가 안 올라
   **자동 롤백**된다. 이것이 "단일 점수를 Eval이 소유"하는 이유다 — 하나의 점수가
   트레이드오프를 내부화해 심판 역할을 한다.
2. **유한한 점수 → 단조 수렴**: overall_score는 [0,1]로 상한이 있고 "오른 것만 유지"
   하므로 끝없이 올릴 수 없다 → 생산적 무한루프는 수학적으로 불가능.
3. **블랙리스트**: 롤백된 `(label, prescription_id)`는 재시도 금지 → 같은 수 반복 차단.
4. **max_iterations**: 하드 스톱.

해결 불가능한 경우의 종착지: 어떤 knob으로도 overall이 안 오르면 → 전부 롤백 → 예산
소진 → **best-so-far로 Serve + "이 문제들은 자동으로 못 고쳤다"고 리포트**. 무한루프가
아니라 정직한 포기.

> **planner가 지켜야 할 것**: top_k를 recall만 보고 판정하지 말 것. 판정은 항상 단일
> overall_score로 하며, 이 원칙을 지키는 한 진동은 구조적으로 막힌다.

---

## 8. 필요한 선행 작업 (의존성)

- **Eval 수정 (near-term, top_k용)**: `_gold_in_wider_candidates`가 **순위**를 반환하게
  하고, `low_rank`/`missing_gold` Finding의 `metadata`에 `{"gold_rank": N, "recall_at_k": r}`를
  싣는다. (`DiagnosticReport.findings_summary`가 이미 "Optimize 소비용"으로 예약돼 있어
  채널은 있음.)
- **Eval 수정 (phase 2, chunk용)**: RAGAS probe에 `gold_spans`(원문 절대좌표)를 채운다
  (현재 no-op). + chunking 진단 신호(under/overchunking) 추가. 이게 없으면 chunk 근거값을
  계산할 데이터 자체가 없다.
  > 진행 방식: **Eval 코드는 이승준이 직접 수정하고, main으로 PR을 올려 Eval 파트가
  > 리뷰·확인하는 방식으로 합의됨.**
- **Index — 확인 완료, 수정 불필요** ✅: 코드 확인 결과 Index는 이미
  (1) `index_config`의 `chunk_size`/`chunk_overlap`을 읽어 **실제로 재청킹**하고,
  (2) `char_span`·`chunk_index`·`token_count`를 **모두 생성**한다.
  즉 Optimize의 chunk 처방은 실제로 반영되며, Eval의 `gold_spans` 재동기화에 필요한
  재료도 이미 준비돼 있다. **chunk 작업의 병목은 Index가 아니라 Eval 쪽뿐이다.**

## 9. 역할 분배 (합의·구현 완료)

### 9-1. 나누는 기준 — 이미 있는 planner ↔ optimizer 경계

파라미터 튜닝은 성격이 다른 두 일로 쪼개진다. 이 경계는 **새로 만드는 게 아니라 지금
코드에 이미 있는 계층 분리**이며, 각자 이미 소유한 파일과 그대로 일치한다.

```
[1] "top_k를 11로 하자"고 계산하는 일        → planner.py   (무엇을 시도할지)
[2] "11이 안전한지 확인하고 실제로 적용"      → optimizer.py (어떻게 탐색·검증할지)
```

### 9-2. 담당

| 담당 | 파일 | 할 일 |
|------|------|-------|
| **이승준** | `planner.py`<br>(+ `agents/eval/*`) | ① 측정값 → 목표값 **계산** (gold_rank → top_k)<br>② **probe 집계** (백분위/커버리지, max 금지)<br>③ **`confirmed` 게이팅** (예비 진단엔 비싼 trial 금지)<br>④ Eval 수정 (gold_rank 노출, phase2의 gold_spans) |
| **권성우** | `optimizer.py` | ⑤ **`internal` backend** = 방문 간 **sweep 상태 기계**<br>(어느 축을 몇 번째 후보까지 쓸어봤고 점수가 뭐였나)<br>⑥ **안전범위·capability 강제** (top_k ≤ 20 캡 등)<br>⑦ 결과 정규화 (`OptimizationResult`) |

**자연스러운 핸드오프**: planner가 근거값 후보를 만들면(이상치 200 포함 가능) →
optimizer의 안전범위 필터가 캡·거부한다. **이건 이미 코드가 그렇게 동작하므로
(§4-1 top_k ≤ 20) 새로 협의할 것이 없다.**

### 9-3. 실제로 어떻게 맞물렸나

각자 자기 파일에서 병렬로 구현한 뒤 병합했고, 두 조각이 정확히 맞물렸다.

```
성우: "후보 목록이 어디선가 온다"고 가정하고 sweep 배관을 만듦
      (internal backend, trial_results, chunk_prescreener, baseline metrics)
                          ↕
승준: 그 후보를 만드는 두뇌를 만듦
      (라벨 묶음 → 무릎 분석 → 후보 [8,12,15])
```

후보 전달은 `_finding_search_space` 의 **3단 우선순위**로 정리됐다.

1. `Finding.metadata["parameter_candidates"]` — Eval 이 직접 후보를 줄 때의 확장점
   (성우님이 남겨둔 자리. 후보 산출을 Eval 로 옮겨도 planner 를 안 고치게 하는 목적)
2. `_GROUNDED_VALUES` — planner 가 진단 측정값에서 계산한 근거값 (승준)
3. 방향 키워드 `×2` 추측 — 근거가 없을 때의 최후 폴백 (기존)

**3층 분류(§3)는 여기서 폐기됐다.** planner 는 `{경로: [후보]}` 하나의 형식으로만 주고,
실행 정책은 optimizer 가 경로별로 고른다 — 분류라는 개념 자체가 필요 없었다.

### 9-4. 남은 협의 지점

- **`_MAX_STEP_PER_PROBE` / `_MAX_SWEEP_CANDIDATES` 튜닝** — 지금은 각각 2.0 / 3.
  실제 파이프라인 실험 데이터로 보정 필요.
- **`gold_rank` Eval 계약** (§8) — `low_rank`/`missing_gold` 근거화의 전제.

## 11. 이 방식(1안)의 요약과 대안

- **1안 (이 문서)**: 진단 측정 → 값 계산(되는 라벨) + 소수 sweep(안 되는 라벨), 우리
  루프가 주도. 장점: 정확·저비용·설명 가능. 한계: chunk 근거화에 Eval/Index 선행 작업 필요.
- **대안 A (Optuna ask-and-tell)**: 나중에 **여러 파라미터를 동시에(joint)** 튜닝하고
  싶어지면 고려. 단 평가가 비싸고 적은 현 상황에선 표본 부족으로 이점이 작다.
- **대안 B (RAGBuilder 유지)**: 제어권 방향 문제로 기각(1절).

> **정리**: 단일 축 좌표하강 + 진단 근거 값 계산이 우리 아키텍처·비용·제품 철학에 가장
>잘 맞는다. 우선 top_k부터 근거화하고 chunk는 데이터가 준비되면 확장한다.
